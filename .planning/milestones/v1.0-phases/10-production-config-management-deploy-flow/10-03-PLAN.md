```markdown
## Config Updates on Existing Installs

**New workflow (Phase 10):** The entrypoint now uses `drush deploy` for existing installs. This runs `updb -> cr -> cim -> cr -> deploy:hook` — the standard Drupal config sync workflow.

- **Config source of truth:** `config/sync/` directory (1053+ YAML files, committed to git)
- **Settings.php:** Template committed to git with `getenv()` calls (NOT generated by entrypoint)
- **Config overrides:** `$config[]` in settings.php for per-instance values (Solr, Qdrant, Gemini)
- **Module exclusions:** `$settings['config_exclude_modules']` excludes demo and web3 modules from config sync. These are enabled per-instance by the entrypoint.

### To deploy config changes:
1. Make changes locally (DDEV)
2. Run `ddev drush cex -y` to export to config/sync/
3. Commit and push to main
4. CI builds new image, Coolify deploys it
5. Entrypoint detects existing install, runs `drush deploy`
6. Config changes are applied incrementally (no volume wipe needed)

### When volume wipe IS still needed:
- Schema changes that `drush updb` cannot handle
- Module removals (uninstall before removing from codebase)
- Corrupt database state
```

**Note:** As of Phase 10, the entrypoint no longer generates settings.php. A template settings.php is committed to git and included in the Docker image. The entrypoint only handles: hash salt generation, service waits, Qdrant provisioning, install detection, drush deploy/site:install, module enables, and Apache start.
```

**3. Remove the old "Config Updates on Existing Installs" content** that references `drush config:import --partial` per module directory. This pattern is now replaced.

```
