# Phase 05.1: Split Related Content Block into Related Topics and Related Events - Research

**Researched:** 2026-02-27
**Domain:** Drupal Block Plugin architecture, Search API query filtering, Open Social content types
**Confidence:** HIGH

## Summary

The current implementation has a single `RelatedContentBlock` (plugin ID: `social_ai_related_content`) placed in the `complementary_bottom` region. It queries the `social_posts` Search API index for semantically similar nodes using vector similarity, with no filtering by content type (bundle). The block is visible on both topic and event node pages via `entity_bundle:node` visibility condition.

The goal is to split this into two separate blocks: **Related Topics** (showing only topic nodes) and **Related Events** (showing only event nodes). This requires filtering the Search API query by the `type` field already present in the `social_posts` index, creating a second block plugin (or parameterizing the existing one), and deploying two block placement configs instead of one.

**Primary recommendation:** Add a `bundle_filter` configuration option to `RelatedContentService::findRelated()`, create two block plugins (or one parameterized plugin with a bundle setting), and replace the single block config with two separate block placement YAMLs.

## Architecture Patterns

### Current Architecture

```
RelatedContentBlock (plugin: social_ai_related_content)
  └── RelatedContentService::findRelated($entity, $account, $limit)
        └── Search API query on social_posts index
              ├── Excludes source node
              ├── Applies permission filters
              ├── Fulltext similarity on rendered_item
              └── Returns NodeInterface[] (NO bundle filter)
```

Block config (`block.block.social_ai_related_content.yml`):
- Region: `complementary_bottom`
- Theme: `socialblue`
- Visibility: `entity_bundle:node` for bundles `topic` and `event`
- Label: "Related Topics" with subtitle "in the community"

### Pattern 1: Parameterized Block Plugin with Bundle Configuration

**What:** Add a `bundle` configuration field to the block plugin so the same PHP class can be instantiated as either "Related Topics" or "Related Events" with different config.

**When to use:** When block behavior is identical except for a filter parameter. This is the standard Drupal pattern.

**How it works:**

1. Add `defaultConfiguration()` and `blockForm()`/`blockSubmit()` to `RelatedContentBlock` with a `bundle` select field
2. Pass the configured bundle to `RelatedContentService::findRelated()`
3. In the service, add a `$query->addCondition('type', $bundle)` when bundle is specified
4. Create two block placement config YAMLs, each with different bundle settings

**Key implementation detail:** The `social_posts` index already has a `type` field (line 77-83 of the index config) that maps to `property_path: type` on `entity:node` datasource. This field contains the node bundle machine name (`topic`, `event`, `post`). Filtering by this field is a simple `addCondition('type', 'topic')`.

### Pattern 2: Two Separate Block Plugins (NOT recommended)

**What:** Create `RelatedTopicsBlock` and `RelatedEventsBlock` as separate classes.

**Why not:** Code duplication. Both blocks would be identical except for the bundle filter value. The parameterized approach is cleaner and more maintainable.

### Anti-Patterns to Avoid

- **Hardcoding bundle in the block class:** Use configuration, not hardcoded values. This allows future bundle additions without code changes.
- **Creating a new service for each bundle:** The existing `RelatedContentService` should handle bundle filtering via a parameter. Do not create `RelatedTopicsService` and `RelatedEventsService`.
- **Removing the existing block config without replacing it:** The old `block.block.social_ai_related_content.yml` must be removed from config/sync and replaced with two new configs. Leaving it causes a stale block placement.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bundle filtering in search | Custom entity query post-filter | `$query->addCondition('type', $bundle)` | Search API handles this natively; post-filtering wastes results and breaks pagination |
| Block configuration forms | Custom settings storage | `BlockBase::defaultConfiguration()` + `blockForm()` | Standard Drupal block config pattern with schema support |
| Theme hook variants | New theme hooks per block type | Same `social_ai_related_content` theme with items | The template just renders items; content type doesn't affect rendering |

## Common Pitfalls

### Pitfall 1: The `type` field value format

**What goes wrong:** Assuming the `type` field contains human-readable names like "Topic" instead of machine names.
**Why it happens:** The `type` property on nodes maps to the bundle plugin ID, which is the machine name.
**How to avoid:** Use machine names: `topic`, `event`, `post`. The index config confirms `type` is a string field from `property_path: type`.
**Warning signs:** Empty results from the related content query.

### Pitfall 2: Forgetting to update hook_preprocess_block subtitle

**What goes wrong:** Both blocks show "in the community" subtitle, or the events block has no subtitle.
**Why it happens:** `social_ai_indexing_preprocess_block()` currently hardcodes the subtitle for plugin ID `social_ai_related_content`.
**How to avoid:** Update the preprocess hook to check the block's configured bundle and set appropriate subtitles, or move subtitle logic into the block plugin's `build()` method.
**Warning signs:** UI labels don't match the block's content type.

### Pitfall 3: Block visibility conditions overlap

**What goes wrong:** "Related Topics" block appears on event pages and vice versa.
**Why it happens:** If both block placements use the same visibility condition (bundles: topic + event), both appear on both page types.
**How to avoid:** Each block config should restrict visibility to its own bundle: Related Topics visible only on topic pages, Related Events visible only on event pages.
**Warning signs:** Duplicate blocks in sidebar showing the same filtered content.

### Pitfall 4: Stale block config not removed

**What goes wrong:** Three blocks appear instead of two (the old combined one plus two new ones).
**Why it happens:** Not removing `config/sync/block.block.social_ai_related_content.yml` when adding the two new configs.
**How to avoid:** Delete the old config file and add two new ones in the same commit.
**Warning signs:** Three related-content blocks in the block layout admin.

### Pitfall 5: Cache tags not including bundle filter

**What goes wrong:** Cached related topics don't invalidate when new topics are created.
**Why it happens:** Cache tags only reference the source node, not a list tag.
**How to avoid:** Add `node_list:topic` or `node_list:event` cache tags based on the configured bundle so the block refreshes when new content of that type is added.
**Warning signs:** Related content stays stale after new content is created.

## Code Examples

### Adding bundle filter to RelatedContentService

```php
// In RelatedContentService::findRelated()
// Add after permission filters, before query execution:
public function findRelated(EntityInterface $entity, AccountInterface $account, int $limit = self::DEFAULT_LIMIT, ?string $bundle = NULL): array {
    // ... existing code ...

    $query = $index->query();

    // Exclude source entity
    $sourceId = 'entity:node/' . $entity->id() . ':' . $entity->language()->getId();
    $query->addCondition('search_api_id', $sourceId, '<>');

    // Apply permission filters
    $this->permissionFilter->applyPermissionFilters($query, $account);

    // Filter by content type bundle if specified
    if ($bundle) {
        $query->addCondition('type', $bundle);
    }

    // ... rest of existing code ...
}
```

### Block plugin with bundle configuration

```php
// In RelatedContentBlock:

public function defaultConfiguration(): array {
    return [
        'bundle' => '',
    ] + parent::defaultConfiguration();
}

public function blockForm($form, FormStateInterface $form_state): array {
    $form['bundle'] = [
        '#type' => 'select',
        '#title' => $this->t('Content type'),
        '#options' => [
            '' => $this->t('All'),
            'topic' => $this->t('Topics'),
            'event' => $this->t('Events'),
        ],
        '#default_value' => $this->configuration['bundle'],
    ];
    return $form;
}

public function blockSubmit($form, FormStateInterface $form_state): void {
    $this->configuration['bundle'] = $form_state->getValue('bundle');
}
```

### Updated preprocess hook

```php
function social_ai_indexing_preprocess_block(array &$variables): void {
    if (($variables['elements']['#plugin_id'] ?? '') !== 'social_ai_related_content') {
        return;
    }
    $bundle = $variables['elements']['#configuration']['bundle'] ?? '';
    switch ($bundle) {
        case 'topic':
            $variables['subtitle'] = t('in the community');
            break;
        case 'event':
            $variables['subtitle'] = t('happening soon');
            break;
        default:
            $variables['subtitle'] = t('in the community');
    }
}
```

## Open Questions

1. **Should "Related Events" only show future events?**
   - What we know: The current RelatedContentService has no date filtering. Events in Open Social have `field_event_date` fields.
   - What's unclear: Whether showing past related events is useful or confusing.
   - Recommendation: Start without date filtering (matches current behavior). Add date filtering as a follow-up if needed. The `social_posts` index does not currently index event date fields, so adding this would require index schema changes.

2. **Should Related Topics show on event pages and vice versa?**
   - What we know: The current block shows all related content regardless of type on both topic and event pages.
   - What's unclear: Whether users viewing an event want to see related topics (cross-type discovery) or only related events (same-type focus).
   - Recommendation: Show only same-type: Related Topics on topic pages, Related Events on event pages. This is the clearest UX and matches the block naming. Cross-type discovery can be added later if desired.

3. **What about the `post` bundle?**
   - What we know: The `social_posts` index includes `post`, `topic`, and `event` bundles. The current block visibility only covers `topic` and `event`.
   - What's unclear: Whether posts should also get a "Related Posts" block.
   - Recommendation: Out of scope for this phase. Posts don't currently show the related content block. Add later if needed.
