**Fix core.extension.yml:**

Edit `config/sync/core.extension.yml` to:

1. **Add `localnodes_platform: 0`** to the `module:` list (in alphabetical order, between `localnodes_content_moderation` and `login_destination` or wherever alphabetically correct). This is CRITICAL — without it, `drush cim` will not know localnodes_platform should be enabled.

2. **Remove `localnodes_demo: 0`** from the `module:` list. This module is excluded via `config_exclude_modules` and enabled per-instance by the entrypoint.

3. **Remove `social_demo: 0`** from the `module:` list. Same reason — it is a dependency of demo modules but should not be in config sync since all demo modules are excluded.

IMPORTANT: Do NOT remove `social_ai_indexing` — that module provides the search indexes and processors and SHOULD be in config/sync.

After editing, verify the file is still valid YAML and the module list is in alphabetical order.

**Add key.key.gemini_api_key.yml:**

Copy the content from `html/modules/custom/localnodes_platform/config/install/key.key.gemini_api_key.yml` to `config/sync/key.key.gemini_api_key.yml`. The content is:

```yaml
langcode: en
status: true
dependencies: {  }
id: gemini_api_key
label: 'Gemini API Key'
description: 'API key for Google Gemini, read from GEMINI_API_KEY environment variable.'
key_type: authentication
key_type_settings: {  }
key_provider: env
key_provider_settings:
  env_variable: GEMINI_API_KEY
key_input: none
key_input_settings: {  }
```

NOTE: The `_core.default_config_hash` is intentionally omitted. When Drupal imports this config, it will compute the hash. Exported config has the hash, but for manually created config sync files it is not required — Drupal handles it gracefully.

**Why social_demo removal is safe:**
`social_demo` provides DemoContent plugins and depends on Open Social core. It does NOT provide permissions to standard roles. Excluding it from config sync means `drush cim` will not try to install or uninstall it — the entrypoint handles demo module state per-instance.

**Why localnodes_platform must be in core.extension:**
This module provides critical config: AI assistant config, block placements, search index config, Solr/Qdrant overrides. If it is missing from core.extension.yml, `drush cim` will try to disable it on existing installs, breaking AI features.

  <name>Task 2: Simplify entrypoint — use drush deploy for existing installs, remove settings.php generation</name>

  <action>
Rewrite `docker/entrypoint.sh` with the following changes:

**1. Remove the entire settings.php generation block** (lines 14-67 in current file). The template settings.php from Plan 01 is now committed to git and included in the Docker image via `COPY --from=builder /app/html /var/www/html/html`. The entrypoint no longer needs to generate or modify settings.php.

However, keep the `DRUPAL_HASH_SALT` auto-generation block (lines 8-11) since that generates a random salt at container start if none is provided. But change it to write the salt to an env var that the template settings.php reads via `getenv('DRUPAL_HASH_SALT')`. Since the template already reads it via getenv, the export on line 9 is sufficient. Keep this block as-is.

**2. Replace the existing-install path** (the `else` branch starting at "=== EXISTING INSTALL DETECTED ==="). Replace the entire block with:

```bash
  echo "=== EXISTING INSTALL ==="

  # Standard Drupal deploy: updb -> cr -> cim -> cr -> deploy:hook
  echo "Running drush deploy..."
  $DRUSH deploy -y

  # Ensure instance-specific modules are enabled (excluded from config sync)
  DEMO_MODULE="${DEMO_MODULE:-localnodes_demo}"
  if [ "$DEMO_MODULE" != "none" ]; then
    echo "Ensuring demo module enabled: $DEMO_MODULE..."
    $DRUSH en "$DEMO_MODULE" -y 2>/dev/null || true
  fi

  # Enable Web3 modules (excluded from config sync, enabled per-instance)
  echo "Ensuring Web3 modules enabled..."
  $DRUSH en siwe_login safe_smart_accounts group_treasury social_group_treasury -y 2>/dev/null || true

  echo "=== EXISTING INSTALL READY ==="
```

Key changes from current existing-install path:
- **REMOVED:** The `config:import --partial` loop over module config/install directories (replaced by `drush deploy` which does full `drush cim` from config/sync)
- **REMOVED:** `search-api:index` and `drush cron` on the existing-install path (avoids unnecessary Gemini API costs on every restart — search-api tracks changes organically)
- **ADDED:** `drush deploy -y` which runs: updb -> cr -> cim -> cr -> deploy:hook (the standard Drupal deploy workflow)
- **KEPT:** Module enables for excluded modules (demo + web3)

**3. Modify the fresh-install path** to add `drush deploy` after initial setup. After the `$DRUSH en localnodes_platform -y` line and BEFORE the demo module section, add:

```bash
  # Import full config from sync to align with repo state
  echo "Aligning config with repo state..."
  $DRUSH deploy -y || echo "WARNING: drush deploy had issues (may be normal on first install)"
```

This ensures a fresh install's active config matches config/sync before demo content is loaded. The `|| echo` fallback is needed because `drush deploy` may warn about config differences on a brand new install (e.g., the install profile's config vs config/sync).

**4. Remove the `drush en localnodes_platform -y` from the existing-install path** since it is now in core.extension.yml and `drush deploy` (which runs `drush cim`) will enable it automatically.

**5. Keep everything else unchanged:** service waits, Qdrant collection setup, file permissions, Apache start.

The final entrypoint structure should be:
```
#!/bin/bash
set -e
[SITE_URI and DRUSH setup]
[Hash salt auto-generation]
[Wait for MariaDB, Solr, Qdrant]
[Qdrant collection setup]
[File permissions]
[Detect install state]
if FRESH:
  drush site:install social ...
  drush en localnodes_platform -y
  drush deploy -y  (align with config/sync)
  [demo module enable + content loading]
  [search indexing + cron]
  [web3 module enable]
  [SIWE domain config]
else EXISTING:
  drush deploy -y
  [demo module enable]
  [web3 module enable]
fi
[Final file permissions]
[drush cr]
[exec apache2-foreground]
```
