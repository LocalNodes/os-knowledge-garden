# Phase 10: Production Config Management & Deploy Flow - Research

**Researched:** 2026-03-02
**Domain:** Drupal 10.6 configuration management, containerized deployment, install profile interactions
**Confidence:** HIGH

## Summary

This phase transitions the project from a "wipe and reinstall" deployment model to a proper `drush deploy` workflow where config changes are deployed incrementally to existing instances. The current entrypoint has significant bandaids (partial config imports per module directory, full search reindex on every restart) that need to be replaced with Drupal's standard config sync flow.

The core architectural challenge is that Open Social has a `hook_install()` in its profile, which means you cannot use "install from existing configuration" (`drush si --existing-config`). However, the standard post-install workflow (`drush deploy` = `updb + cr + cim + cr + deploy:hook`) works fine. The project already has a `config/sync/` directory with 1053 exported config files tracked in git, but the Docker image does not copy this directory into the container, and the settings.php `config_sync_directory` setting currently points to a files directory volume rather than the repo-based sync path.

**Primary recommendation:** Ship a template `settings.php` with `getenv()` calls (committed to git), point `config_sync_directory` to the repo-based `config/sync/`, add `config/sync/` to the Docker image, and replace the entrypoint's existing-install path with `drush deploy`. Use `$config[]` overrides in settings.php for all per-instance variation (Solr host, Qdrant host, API keys, site name). Use `config_exclude_modules` to keep demo modules out of config sync. Remove the search reindex-on-every-restart.

## Standard Stack

### Core
| Tool | Version | Purpose | Why Standard |
|------|---------|---------|--------------|
| `drush deploy` | Drush 13.7.1 | Single command for `updb + cr + cim + cr + deploy:hook` | Community standard since Drush 10.3; handles ordering correctly |
| `config/sync/` | Drupal core | Full site config export/import directory | Core CMI workflow; the only config flow that guarantees consistency |
| `$config[]` overrides | Drupal core | Per-instance env-specific values in settings.php | Runtime layer on top of active config; never written to DB, never exported |
| `$settings['config_exclude_modules']` | Drupal core (8.8+) | Exclude demo/dev modules from config export | Avoids config_split complexity for simple module exclusion |

### Supporting
| Tool | Version | Purpose | When to Use |
|------|---------|---------|-------------|
| `hook_deploy_NAME()` | Drush 13.x | One-time imperative code that runs AFTER config import | When you need to programmatically update content/state after new config lands |
| Key module (env provider) | Already installed | Store API keys via env vars, not in config | Already in use for GEMINI_API_KEY; keeps secrets out of config/sync |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `config_exclude_modules` | Config Split | Overkill -- all per-instance differences are env-var-driven; no environment-specific config objects needed |
| `config_ignore` | Manual exclusions | Not needed -- nothing should be manually edited on deployed instances that differs from config/sync |
| `drush deploy` | Manual `updb && cim && cr` | Loses deploy:hook ordering guarantees and the `--no-cache-clear` optimization |
| Entrypoint-generated settings.php | Template settings.php in repo | Generated settings.php is unconventional and fragile; template with `getenv()` is 12-factor standard |

### Pattern 1: Template settings.php with getenv()
**What:** Ship settings.php in git with all environment-specific values read from env vars via `getenv()`. No runtime generation.
**When to use:** Always. This is the 12-factor app standard for Drupal.
**Example:**
```php
// Source: Drupal community best practice, Lullabot ADR, geerlingguy/drupal-container
$databases['default']['default'] = [
  'database' => getenv('DB_NAME') ?: 'opensocial',
  'username' => getenv('DB_USER') ?: 'opensocial',
  'password' => getenv('DB_PASSWORD') ?: 'changeme',
  'host' => getenv('DB_HOST') ?: 'mariadb',
  'port' => getenv('DB_PORT') ?: '3306',
  'driver' => 'mysql',
  'prefix' => '',
];

$settings['hash_salt'] = getenv('DRUPAL_HASH_SALT') ?: 'change-me-in-production';
$settings['config_sync_directory'] = '../config/sync';
$settings['file_private_path'] = '/var/www/private';

// Per-instance config overrides (runtime layer, not stored in DB)
$config['search_api.server.social_solr']['backend_config']['connector_config']['host'] = getenv('SOLR_HOST') ?: 'solr';
$config['search_api.server.social_solr']['backend_config']['connector_config']['port'] = getenv('SOLR_PORT') ?: '8983';
$config['search_api.server.social_solr']['backend_config']['connector_config']['core'] = 'drupal';
$config['search_api.server.social_solr']['backend_config']['connector_config']['path'] = '/';

$config['ai_vdb_provider_qdrant.settings']['host'] = getenv('QDRANT_HOST') ?: 'qdrant';
$config['ai_vdb_provider_qdrant.settings']['port'] = (int)(getenv('QDRANT_PORT') ?: '6333');

$config['key.key.gemini_api_key']['key_provider_settings']['env_variable'] = 'GEMINI_API_KEY';

// Reverse proxy (Traefik behind Coolify)
if (getenv('DRUPAL_REVERSE_PROXY') === 'true') {
  $settings['reverse_proxy'] = TRUE;
  $settings['reverse_proxy_addresses'] = ['0.0.0.0/0'];
  $settings['reverse_proxy_trusted_headers'] =
    \Symfony\Component\HttpFoundation\Request::HEADER_X_FORWARDED_FOR |
    \Symfony\Component\HttpFoundation\Request::HEADER_X_FORWARDED_HOST |
    \Symfony\Component\HttpFoundation\Request::HEADER_X_FORWARDED_PORT |
    \Symfony\Component\HttpFoundation\Request::HEADER_X_FORWARDED_PROTO;
}

// Trusted hosts from env
$fqdn = getenv('SERVICE_FQDN_OPENSOCIAL');
if ($fqdn) {
  $host = preg_replace('#^https?://#', '', $fqdn);
  $settings['trusted_host_patterns'][] = '^' . preg_quote($host, '/') . '$';
}
$settings['trusted_host_patterns'][] = '^localhost$';

// Exclude demo modules from config sync (they differ per instance)
$settings['config_exclude_modules'] = ['localnodes_demo', 'boulder_demo', 'portland_demo', 'social_demo'];

// DDEV integration
if (getenv('IS_DDEV_PROJECT') == 'true' && file_exists(__DIR__ . '/settings.ddev.php')) {
  include __DIR__ . '/settings.ddev.php';
}
```

### Pattern 2: Simplified Entrypoint (Fresh vs Existing)
**What:** The entrypoint handles two paths: fresh install (`drush site:install`) and existing install (`drush deploy`).
**When to use:** Docker container startup.
**Example:**
```bash
# Source: Drupal community pattern for containerized deployments
if [ "$INSTALLED" = false ]; then
  # Fresh install: standard site-install + module enable + demo content
  $DRUSH site:install social ...
  $DRUSH en localnodes_platform -y
  # ... demo content loading ...
  $DRUSH search-api:index
else
  # Existing install: standard deploy flow
  $DRUSH deploy
  # Module enable for instance-specific modules not in config/sync
  $DRUSH en "$DEMO_MODULE" -y 2>/dev/null || true
  $DRUSH en siwe_login safe_smart_accounts group_treasury social_group_treasury -y 2>/dev/null || true
fi
```

### Pattern 3: config_exclude_modules for Demo Module Switching
**What:** Demo modules (localnodes_demo, boulder_demo, portland_demo) differ per instance. Exclude them from config sync so `drush cim` does not try to enable/disable them.
**When to use:** When the same codebase serves multiple instances with different modules.
**Example:**
```php
// In settings.php
$settings['config_exclude_modules'] = [
  'localnodes_demo',
  'boulder_demo',
  'portland_demo',
  'social_demo',
];
```
**Important:** This means `core.extension.yml` in config/sync will NOT list these modules. The entrypoint enables the correct one per instance.

### Pattern 4: Export Config After Fresh Install
**What:** After any fresh install with all modules enabled, export config to update config/sync.
**When to use:** During development when adding new features/config.
**Example:**
```bash
# On DDEV locally:
ddev drush cex -y
# Review changes in config/sync/, commit to git
```

### Anti-Patterns to Avoid
- **`drush config:import --partial` per module directory:** Blunt instrument that overwrites admin customizations and doesn't respect dependencies. Replace with proper `drush cim` from config/sync.
- **`search-api:reset-tracker` on every restart:** Forces full re-embedding through Gemini API ($$$). The `--uri` flag on the initial index already handles URL correctness. Remove entirely from existing-install path.
- **Generating settings.php at runtime:** Fragile, hard to audit, not version-controlled. Ship a template instead.
- **Putting module config in `config/install/` for existing-install updates:** `config/install/` is ONLY read on first module enable. For ongoing config management, config/sync is the mechanism.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Deploy orchestration | Custom entrypoint deploy logic | `drush deploy` | Handles updb/cim/cr ordering, deploy hooks, error handling |
| Per-instance config | Environment-specific YAML files | `$config[]` overrides in settings.php | Runtime-only layer, never exported, never conflicts with config sync |
| Module exclusion from sync | Manual config file editing | `$settings['config_exclude_modules']` | Core feature since 8.8, handles export/import correctly |
| API key management | Hardcoded keys in config | Key module with env provider | Already in use; keeps secrets out of version control |
| Settings.php templating | Entrypoint heredoc generation | Committed settings.php with getenv() | Standard pattern, auditable, version-controlled |
| Config update for distributions | `drush config:import --partial` per dir | Full `drush cim` from config/sync | Respects dependencies, handles deletions, atomic |

**Key insight:** Every "bandaid" in the current entrypoint exists because the proper config sync workflow was not set up. Once config/sync is the source of truth and `drush deploy` handles updates, all the partial imports, manual config sets, and per-module directory imports become unnecessary.

## Common Pitfalls

### Pitfall 1: Open Social hook_install() Blocks Install-from-Config
**What goes wrong:** Attempting `drush si --existing-config` with the `social` profile fails because `social_install()` exists.
**Why it happens:** Drupal core does not allow profiles with `hook_install()` to use install-from-config.
**How to avoid:** Do NOT try to install from config/sync. Always use `drush site:install social` for fresh installs. Config/sync is only for post-install updates via `drush cim` / `drush deploy`.
**Warning signs:** Error "The selected profile has a hook_install() implementation and therefore can not be installed from configuration."

### Pitfall 2: Config Sync Tries to Remove Instance-Specific Modules
**What goes wrong:** `drush cim` sees that boulder_demo is enabled in the database but not in `core.extension.yml` and tries to uninstall it.
**Why it happens:** Config sync enforces the exact module list from core.extension.yml.
**How to avoid:** Use `$settings['config_exclude_modules']` to exclude all demo modules and web3 modules from the sync process. These are enabled per-instance by the entrypoint.
**Warning signs:** `drush cim` showing module uninstall operations you didn't expect.

### Pitfall 3: config_exclude_modules Removes Role Permissions on Export
**What goes wrong:** Since Drupal 9.3, user roles have dependencies on the module providing each permission. If a module is excluded and provides permissions, those role configs get deleted on export.
**Why it happens:** Drupal's config dependency system considers excluded modules as "missing."
**How to avoid:** Only exclude modules that do NOT provide permissions used by core roles. Demo modules (content only, no permissions) are safe. If web3 modules provide permissions to standard roles, handle those permissions via `hook_install()` in localnodes_platform, not in role config YAML.
**Warning signs:** User role YAML files being deleted or modified after `drush cex`.

### Pitfall 4: search-api:reset-tracker Forces Expensive Re-embedding
**What goes wrong:** Running `search-api:reset-tracker` + `search-api:index` on every container restart re-generates ALL vector embeddings via Gemini API, costing money and time.
**Why it happens:** The original intent was to fix URLs (http://default/node/X) but `reset-tracker` marks ALL items for reindex including vector embeddings.
**How to avoid:** Remove the reset-tracker + reindex from the existing-install path entirely. The `--uri` flag on `drush` already ensures correct URLs. If URL fixes are needed one-time, do it via a deploy hook.
**Warning signs:** Gemini API costs spiking on every deployment; 3+ minute deploy times from re-embedding.

### Pitfall 5: Config Sync Conflicts with Open Social Update Hooks
**What goes wrong:** Open Social's `hook_update_N()` modifies config, then `drush cim` overwrites those changes with the config/sync version.
**Why it happens:** `drush deploy` runs `updb` BEFORE `cim`. If an update hook changes config, `cim` will revert it.
**How to avoid:** After running `drush updb` locally during an Open Social update, always run `drush cex` to capture the update hook's config changes into config/sync before committing. The deploy workflow is: update locally -> updb -> cex -> commit -> deploy.
**Warning signs:** Config changes from Open Social updates being silently reverted on deploy.

### Pitfall 6: Missing key.key.gemini_api_key in config/sync
**What goes wrong:** The Gemini API key entity exists in `localnodes_platform/config/install/` but was not exported to `config/sync/`. After a `drush cim`, the key entity might be missing if it was never properly exported.
**Why it happens:** The key was created by module enable, but the export may have been done before the module was enabled, or the config was somehow missed.
**How to avoid:** Ensure a complete `drush cex` captures ALL active config including key entities. Verify key.key.gemini_api_key.yml exists in config/sync.
**Warning signs:** Gemini API calls failing after deploy; "key not found" errors.

## Code Examples

### Simplified Entrypoint Structure
```bash
#!/bin/bash
# Source: Drupal community pattern for containerized deployments
set -e

SITE_URI="${SERVICE_URL_OPENSOCIAL:-http://localhost}"
DRUSH="/var/www/html/vendor/bin/drush -r /var/www/html/html --uri=$SITE_URI"

# --- Wait for services ---
echo "Waiting for MariaDB..."
while ! nc -z "${DB_HOST:-mariadb}" "${DB_PORT:-3306}" 2>/dev/null; do sleep 1; done

echo "Waiting for Solr..."
while ! nc -z "${SOLR_HOST:-solr}" "${SOLR_PORT:-8983}" 2>/dev/null; do sleep 1; done

echo "Waiting for Qdrant..."
while ! bash -c "echo > /dev/tcp/${QDRANT_HOST:-qdrant}/${QDRANT_PORT:-6333}" 2>/dev/null; do sleep 1; done

# --- Qdrant collection setup (infrastructure, not app config) ---
# [keep existing Qdrant collection provisioning logic]

# --- File permissions ---
chown -R www-data:www-data /var/www/html/html/sites/default/files 2>/dev/null || true
chown -R www-data:www-data /var/www/private 2>/dev/null || true

# --- Detect install state ---
INSTALLED=false
if $DRUSH status --field=bootstrap 2>/dev/null | grep -q 'Successful'; then
  INSTALLED=true
fi

if [ "$INSTALLED" = false ]; then
  echo "=== FRESH INSTALL ==="
  $DRUSH site:install social \
    --site-name="${DRUPAL_SITE_NAME:-LocalNodes}" \
    --account-name="${DRUPAL_ADMIN_USER:-admin}" \
    --account-pass="${DRUPAL_ADMIN_PASS:-admin}" \
    --account-mail="${DRUPAL_ADMIN_EMAIL:-admin@example.com}" \
    --db-url="mysql://${DB_USER:-opensocial}:${DB_PASSWORD:-changeme}@${DB_HOST:-mariadb}:${DB_PORT:-3306}/${DB_NAME:-opensocial}" \
    -y

  # Enable platform module (brings in AI config, blocks, etc.)
  $DRUSH en localnodes_platform -y

  # Import full config from sync to align with repo state
  $DRUSH deploy

  # Enable instance-specific demo module
  DEMO_MODULE="${DEMO_MODULE:-localnodes_demo}"
  if [ "$DEMO_MODULE" != "none" ]; then
    $DRUSH en "$DEMO_MODULE" -y
    # [demo content loading logic]
  fi

  # Enable Web3 modules
  $DRUSH en siwe_login safe_smart_accounts group_treasury social_group_treasury -y || true

  # Index content
  $DRUSH search-api:index
  $DRUSH cron && sleep 5 && $DRUSH cron && sleep 5 && $DRUSH cron

  echo "=== FRESH INSTALL COMPLETE ==="
else
  echo "=== EXISTING INSTALL ==="
  # Standard Drupal deploy: updb -> cr -> cim -> cr -> deploy:hook
  $DRUSH deploy

  # Ensure instance-specific modules are enabled (excluded from config sync)
  DEMO_MODULE="${DEMO_MODULE:-localnodes_demo}"
  if [ "$DEMO_MODULE" != "none" ]; then
    $DRUSH en "$DEMO_MODULE" -y 2>/dev/null || true
  fi
  $DRUSH en siwe_login safe_smart_accounts group_treasury social_group_treasury -y 2>/dev/null || true

  echo "=== EXISTING INSTALL READY ==="
fi

# Final file permissions
chown -R www-data:www-data /var/www/html/html/sites/default/files 2>/dev/null || true
chown -R www-data:www-data /var/www/private 2>/dev/null || true

# Start Apache
exec apache2-foreground
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `config/install/` per module for updates | `config/sync/` + `drush deploy` | Drupal 8.0+ (config sync), Drush 10.3 (deploy command) | Atomic, dependency-aware config deployment |
| `drush config:import --partial` | `drush config:import` (full) | Always was the standard | Partial import ignores deletions, breaks dependency tracking |
| Features module for distributions | `hook_update_N()` + config_update + update_helper | Open Social moving away from Features since 2020 | Cleaner update path |
| Generating settings.php at runtime | Template settings.php with `getenv()` | 12-factor app methodology | Auditable, version-controlled, standard |
| Config Split for env differences | `$config[]` overrides in settings.php + `config_exclude_modules` | Lullabot ADR 2021, growing community consensus | Simpler, no contrib module dependency for basic env differences |

**Deprecated/outdated:**
- `$config_directories['sync']`: Replaced by `$settings['config_sync_directory']` in Drupal 8.8+
- Features module for config management: Open Social actively moving away from it
- `drush config:import --partial --source=DIR`: Bandaid that doesn't respect dependencies or deletions

## Open Questions

1. **Web3 modules in config_exclude_modules**
   - What we know: Web3 modules (siwe_login, safe_smart_accounts, group_treasury, social_group_treasury) are not currently in config/sync's core.extension.yml. They are enabled by the entrypoint.
   - What's unclear: Do these modules provide permissions that are assigned to standard roles? If so, excluding them could cause role config to be deleted on export.
   - Recommendation: Check if web3 modules add permissions to existing roles. If yes, handle those permissions in localnodes_platform's hook_install() (already done for 'access deepchat api'). If no, safely add them to config_exclude_modules.

2. **Complete config export needed**
   - What we know: config/sync has 1053 files but is missing key.key.gemini_api_key.yml. There may be other gaps.
   - What's unclear: When was the last full export? Are there other missing configs?
   - Recommendation: Do a clean fresh install, enable all modules, run `drush cex`, and diff against current config/sync. This is a prerequisite task for this phase.

3. **DDEV settings.php interaction**
   - What we know: DDEV generates settings.ddev.php which overrides database connection. The current settings.php includes it at the bottom.
   - What's unclear: Will the new template settings.php conflict with DDEV's settings.ddev.php?
   - Recommendation: Keep the DDEV include at the bottom of settings.php. DDEV's settings.ddev.php overrides `$databases` and adds its own config, which is fine since it loads last. The template settings.php should have DDEV defaults that get overridden.
